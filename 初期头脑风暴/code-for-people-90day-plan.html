零、整体框架

13 周分三阶段,每阶段 4-5 周。每阶段结束有一个明确的判断点:

阶段周次核心任务结束时判断点
阶段 1
田野		 Week 1-5 3-5 个并行 CD 轨道,选定 1-2 个 MVP 方向 有没有发现真痛点?痛点对应工具能不能 1-2 周建出来?
阶段 2
建造		 Week 6-9 第一个 MVP 完整开发 + 备案 + 数据合规 能不能上线?有没有把法律雷区都拆完?
阶段 3

5 周田野调查识别真痛点,4 周建一个能上线的微信小程序,4 周用 50 个种子用户验证——结束时拿到一个清晰的 continue/pivot/kill 决策,以及一份足以指导下一个 90 天的复盘。

一、选题方法论

1.1 三条筛选原则(同时满足)

  1. 目标用户是第一/二/服务业劳动者(v2.1 核心承诺,不可妥协)
  2. 3 人小队 + AI 能在 4 周内做出可用 MVP(否则 90 天兜不住一个完整循环)
  3. 用户能(也愿意)在 6 个月内付费 ≥ 10 元/月,或贡献等价的内容/数据(否则商业模型不闭环,养不起项目)

1.2 候选轨道(田野前的初步清单)

v2.1 没限定具体方向。下面 5 个候选都满足上面三条筛选,前 5 周田野并行展开,Week 5 末选 1-2 个进 Phase 2:

候选方向用户可能的 MVP风险
外卖骑手收入/路径助手 美团/饿了么/达达骑手 AI 帮算"这单值不值"、"这片区今晚补贴预测"、收入流水自动整理 跟美团抢饭碗易被封号,数据合规复杂
新手妈妈生活助手 0-1 岁宝妈,主要县城/三四线 AI 问诊预筛(发烧/起疹/吐奶判轻重)+ 二手母婴流转 + 月嫂vetting 儿童信息合规雷区;医疗类内容审查严
电子厂工人生活工具 富士康/立讯/比亚迪等流水线工人 工时/加班/工资自动核对、宿舍/食堂/兼职信息汇总、AI 帮写请假/投诉信 用户碎片时间少,留存难;工厂方可能反感
家政/月嫂/育儿嫂工具 家政从业者(供给侧) 客户管理 + 自动账单 + 雇主沟通模板 + 接单口碑 从业者数字化程度差异大,小白用户教育成本高
小餐馆/夫妻店经营助手 30 平米以下个体经营者 AI 帮写菜单文案、采购清单、社区团购对接、给小红书发宣传 竞品多(美团商家版等),价值差异化弱
备注

这 5 个里面我把 外卖骑手 和 新手妈妈 标为最优先,因为你已经显示出对这两个的倾向(注册骑手 + b 超室门口聊天)。但 Week 1-4 期间至少同时跑 3 个轨道,不要早期 lock-in 一个。

1.3 田野方法:三三制 + 24h demo + 真金白银

从 v2.0 继承,在此具体化:

  1. 三三制:每个轨道每周接触 3 类用户(典型/边缘/反例),每类 3 个人,共 9 人。每周复盘是"9 个故事 → 1 个抽象痛点"
  2. 24h demo:每个候选轨道,在田野第二周必须有一个"24 小时内能跑通核心流程的 demo"(不需要好看,需要可见)。让用户看着 demo 评论,比让他描述需求更准
  3. 真金白银:每次访谈给受访者 50-100 元红包或等价实物。这件事是设计 feature 不是 nice-to-have——付费的访谈数据质量比免费的高 3 倍以上(沉没成本让受访者愿意"出力"),而且 50 元对你便宜,对工友是有意义的

二、田野调查具体安排

你点名了三个具体动作——注册骑手、应聘电子厂、b 超室门口聊天。我把每个动作的实操细节、时间投入、隐藏风险全部摊开,你看完决定是否照执行。

2.1 注册外卖骑手(Sun 主要承担)

实操流程

  1. 选平台:美团骑手最大(单量多)、饿了么蜂鸟其次、顺丰同城/达达再次。建议美团——数据样本最有代表性
  2. 下载美团骑手 App,选择"众包"类型(不是"专送")。众包没有班次约束,Sun 想跑就跑
  3. 实名认证:身份证 + 拍证件照
  4. 办健康证:大部分城市强制,当地疾控中心或指定医院,抽血 + 胸透,100-200 元,1-3 天出证
  5. 看视频培训 + 考试:1-2 小时内完成
  6. 开始跑单

时间投入建议

不少于 2 周,每周 ≥ 20 小时实际跑单(主要在中午 11-13、晚上 17-20 高峰)。少于这个量,你只算"体验过",不算"真理解骑手"。

核心目的(优先级排序)

  1. 在 等单 / 取餐排队 / 充电桩 等场景跟其他骑手聊天 —— 这是金矿,不是跑单本身
  2. 体验 App 算法:跑单价格波动、派单逻辑、补贴规则、惩罚机制
  3. 体验劳动强度 + 风险(电瓶车事故、超时罚款、商家催促)
  4. 顺便挣点跑单费(50 元/次访谈预算的一部分可以从这里出)
需要警觉的事
  • 你不会真的变成骑手。你是"穿骑手皮的程序员",其他骑手大多数会看出来(口音、用词、对超时的反应)。不要装——直接说"我是个程序员想做工具帮骑手,能跟你聊 30 分钟吗?给你 50 块"。"诚实的外人"比"潜入式"高效得多
  • 美团 KOL 化的骑手(已经做小红书/抖音的骑手)不是好样本。他们看世界的方式更接近自媒体而不是普通骑手。专门找"什么自媒体都不做的骑手"
  • 电瓶车安全。北京/杭州冬天跑骑手摔车风险很真实,买好头盔,雨天别跑

2.2 应聘电子厂(学徒主要承担,Sun 配合)

实操流程

  1. 找派遣公司:在 BOSS 直聘 / 智联 / 58 同城搜"电子厂 普工 短期",大量派遣公司发单。也可以去富士康直招通道(郑州/深圳)
  2. 选短期合同/小时工:3-7 天试工免费,7 天后转正可走可留。这个机制对你刚好——目的是访谈不是真打工
  3. 体检:进厂前体检,大多数派遣公司负担费用
  4. 进厂:住宿舍 + 上流水线

时间投入建议

学徒进厂 7-10 天(身体能承受、信息收集足够)。Sun 不建议进厂——一个 30+ 的男性中级程序员进流水线显眼,反而妨碍学徒做田野。Sun 在外配合(每天晚上跟学徒通话复盘 + 帮整理笔记)。

核心目的

  1. 宿舍生活:8 人间、谁玩什么手机 app、晚上聊什么、对加班的态度
  2. 流水线日常:工时、计件 vs 计时、罚款规则、跟班长的关系
  3. 食堂 + 班车:聊天最高频场景。学徒带本子记录(不要在工友面前拿手机录音)
  4. 工资条:实际工资构成(底薪 + 加班费 + 满勤奖 + 各种扣)。这是工具方向的金矿——"自动核对工资"是个真痛点
需要警觉的事
  • 学徒不是真打工赚钱的。要在心里清楚他是"研究员",一旦觉得不行(身体撑不住、出现安全风险)立即出来。不要"为了田野咬牙顶"
  • 不要假身份。说"我对工厂生活好奇,想来看看,给我介绍工作的师傅我都告诉他了"。派遣链上有人不喜欢你套话也直接走
  • 女工人/男工人比例:大多数电子厂女工多,学徒是男性进去会被分到男工区(包装/物流/检测),女工的世界他接触不到。如果母婴方向是你的优先,这件事是个 gap——补救:Sun 或学徒的女性家属/朋友去电子厂女工聚集的地方(下班时的厂门口、商业街、宿舍楼下)做访谈

2.3 b 超室门口聊天(改进版)

原方案的问题

妇幼保健院 b 超室门口大量是孕妇 + 陪同的父亲/婆婆/母亲。等候时间长(2-4 小时)、容易开口——你这个直觉对的。但有一个执行问题:一个陌生中年男性蹲在妇幼医院里找孕妇聊天,容易被保安请走,也容易引起警觉。这是结构问题,不是你不善聊。

改进方案(三选一或多管齐下)

  1. 找女性研究员:Sun 的太太/姐妹、学徒的家属、高中同学的太太——任何一个女性愿意帮忙做田野,她蹲妇幼医院门口的合法性 + 信任成本比 Sun 低 10 倍。承诺给她按小时付费(100 元/小时),把她当真正的研究员对待
  2. 换场景:同样的孕妇/新手妈妈群体,在月嫂培训机构 / 月子中心 / 早教中心 / 母婴店更易接触,且这些地方店主/培训师本身就是潜在用户(供给侧)。聊天阻力小
  3. 线上前置:微博/小红书/抖音搜"#新手妈妈" "#月嫂避坑"等标签,找发了很多帖子但粉丝不多的用户私信(她们正在表达但还没找到人听)。约 30 分钟视频访谈 + 50 元红包。这种用户访谈成本极低,质量极高

无论选哪个,核心要问的

  • "从怀孕到孩子 1 岁,你做的所有决定里最难做的是哪个?"
  • "这个决定你最后是问谁的?" (好友/婆婆/医生/抖音/淘宝评论)
  • "那个人给的建议对吗?" 和 "如果是错的,当时谁本来能告诉你正确的?"
  • "每个月你为孩子花的钱大头在哪?" (奶粉/纸尿裤/早教/医疗)
  • "有没有什么事情你想做但没做,因为没钱/没时间/没人帮?"

2.4 第四个轨道:线上扫一遍(高中同学如已入伙,他主导)

方法

在小红书/抖音/快手搜索目标用户群的标签,找已经发声但还没成网红的用户(粉丝 200-2000、内容真实、最近 1 个月有更新)。私信约访,30 分钟视频访谈 + 50 元红包。

典型搜索词

  • 骑手:"#外卖员日常" "#送外卖" "#骑手日记"
  • 新手妈妈:"#新手妈妈" "#一胎宝妈" "#月嫂避坑"
  • 电子厂:"#富士康" "#流水线日记" "#工厂打工"
  • 家政:"#月嫂日记" "#育儿嫂"

这条轨道的特殊价值

面对面田野只能接触你物理上能去的人,但你的产品用户分布在全国。线上能采到 不同城市/省份 的用户,验证地域差异。每个 vertical 至少采到 3-5 个不同城市的样本。

三、技术架构

3.1 一句话原则

3 人小队 + AI + 90 天 + 50 用户 = 用最简单成熟的技术栈,把复杂度推后到真有规模后再处理。所有"将来可能需要"的架构决策都先不做。

3.2 推荐技术栈

层选择理由
前端(主) Taro 4 + React 18 + TypeScript 一套代码同时编译微信小程序 + 抖音小程序 + H5。中国 B2C 工友 90% 通过微信触达,小程序无应用商店审核,分享/支付/登录链路最短
UI 组件库 NutUI-React Taro 版 京东出品、Taro 官方推荐、组件最全(表单/弹层/列表/上拉刷新开箱即用)。比 Taroify 更适合工具型应用
样式 Tailwind CSS(Taro 4 已支持) 跟 sunmer-home 既有 patterns 一致,迁移零成本
状态管理 Zustand 轻量、跨端兼容、学徒友好(不需要学 Redux 思维)
数据层 TanStack Query (React Query) + axios client 跨端可用、缓存策略成熟
后端 NestJS (Node.js) TS 端到端,与 Taro 前端共享类型/逻辑代码最自然。可复用 Payload CMS 部分能力
数据库 PostgreSQL(主)+ Redis(缓存) 关系型够用,JSONB 字段灵活,免去早期 schema 焦虑
对象存储 阿里云 OSS 或 腾讯云 COS 图片/视频/文档存储,境内服务,合规友好
LLM DeepSeek API(主)+ Qwen/Doubao(备) 国内访问稳定、价格低、合规;v2.0 提到的"模型主权"长期路径起点
后端部署 阿里云轻量服务器 ECS + RDS PostgreSQL 试点期 1 台 2c4g + RDS 基础版,月成本 < 300 元。备案前先用境外节点开发,备案完切回
小程序 CI GitHub Actions + miniprogram-ci(微信官方) 自动上传体验版到微信开发者工具,PR 触发预览
CI/CD GitHub Actions(免费 + 团队熟) 不要花时间搭 Jenkins,够用就行
监控 Sentry(错误) + 阿里云内置(基础指标) SaaS 免费层够用

3.3 不要做的事(显式列出)

  • 不要造微服务。单体应用,1 个 git repo,1 个数据库。直到用户超过 5000 都不需要拆
  • 不要做 App(原生 iOS/Android)。商店审核 + 各种 SDK + 推送资质 + 隐私清单 = 6 周时间在合规上,不划算
  • 不要自部署模型(本阶段)。v2.0 模型主权是长期目标,试点期用 API 就够。等用户量 > 1000 再考虑
  • 不要做"中台/低代码平台"。先把第一个产品做出来。多团队的工具放到 v2.1 §7.3 推迟决策清单里
  • 不要追求 100% 测试覆盖。关键链路(支付、登录、数据写入)有冒烟测试就够
  • 不要做用户增长黑客技术。前 50 用户全部线下/线上人肉拉到,不需要 SEO/广告投放

3.4 仓库起点:fork sunmer-home

Sun 既有项目 Stupidism/sunmer-home(Turborepo 多 app monorepo)作为新项目骨架,fork 之后剥离不相关 apps,新建 Taro app:

# 1. Fork + clone
gh repo fork Stupidism/sunmer-home --fork-name code-for-people-app --clone
cd code-for-people-app

# 2. 剥离不相关 apps
rm -rf apps/wedding-invite apps/nunu-island apps/nunu-log

# 3. 新建 Taro 小程序 app
cd apps
pnpm dlx @tarojs/cli init gongyou-mp
# 选项:React + TypeScript + Tailwind CSS

# 4. 安装 NutUI-React Taro 版
cd gongyou-mp
pnpm add @nutui/nutui-react-taro

# 5. 注册微信小程序开发者账号(个人,免费)
# https://mp.weixin.qq.com/  Sun 个人微信注册,试点期用个人账号
# 公司注册之后(Week 5)切企业账号

最终的 monorepo 结构:

code-for-people-app/
├── apps/
│   ├── gongyou-mp/        # Taro 小程序(主力)
│   ├── gongyou-web/       # Next.js 落地页(自媒体导流;bubu-log 改造)
│   └── gongyou-api/       # NestJS 后端
├── packages/
│   ├── ui-mp/             # Taro 端组件(基于 NutUI-React 二次封装)
│   ├── ui-web/            # Web 端组件(继承 sunmer-home 的 shadcn/ui)
│   ├── shared/            # 业务逻辑、类型、API client(跨端共享)
│   └── typescript-config/ # 继承 sunmer-home
├── .github/workflows/
│   ├── miniprogram-deploy.yml  # 小程序自动上传体验版
│   └── vercel-preview.yml      # Web 落地页 PR preview
└── (其他基础设施从 sunmer-home 继承)
关键判断

packages/ui-mp 和 packages/ui-web 拆开两个,不强求"一套组件跨端"。Taro 理论支持但 Web 和小程序的交互模式差异(下拉刷新、原生导航、键盘行为)让"统一组件"得不偿失。业务逻辑共享(packages/shared),UI 各自实现——这是 Taro 多端成熟范式。

3.5 AI 协作约定

3 人小队 + AI 协作的工程产出能达到过去 8-10 人团队,但需要明确约定:

  • 每个人有一个 Claude / DeepSeek / Cursor 的固定工作流。新人入伙第一周内完成自己的 setup
  • 代码 review 不可省。AI 生成的代码必须有人读懂再 merge——盲信 AI 是项目最大的隐藏债务来源
  • 架构决策不让 AI 拍板。AI 适合执行已定型的技术决策,不适合在两个方案间做选择
  • 提示词共享。三个人维护一个内部仓库,把好用的 prompt 模板沉淀下来

四、上线路径

4.1 已锁定:小程序优先(Taro)

不再比较 Web / 小程序 / App 三选一——已决:微信小程序作为主入口,Web 作为自媒体落地页,App 完全不做。

  1. 主要产品:微信小程序(Taro 编译产物)——所有核心功能在这里,目标用户(工友)的默认入口
  2. 导流落地页:H5(Taro 同代码编译)——自媒体内容 + "下面扫码进小程序"的桥
  3. 公众号——内容沉淀 + 关注后的私域池 + 客服

4.2 小程序具体上线流程

  1. 注册小程序账号:个人 / 企业。个人小程序类目受限(不能做电商、不能做支付、不能做互动社区),所以推荐用企业账号
  2. 注册时机的矛盾:v2.1 说 90 天试点不注册公司,但小程序企业账号需要公司。解决路径见下面 §4.4
  3. 小程序 ICP 备案(2024 新规):提交后 7-20 工作日
  4. 类目审核:选错类目会反复打回,提前看好微信小程序类目清单
  5. 开发 → 提审 → 上线

4.3 公司注册时机的现实建议(修订 v2.1)

v2.1 修订点

v2.1 §3.1 阶段 2 的触发条件原本是"用户 ≥ 100"。实际操作中,公司注册要先于此——因为没有公司你跑不通"小程序企业账号 + ICP 备案"这条线。

修订建议:公司注册时间提前到 Week 5 末(田野结束、确定方向之时),不是等到用户 100。原因是:

  • 公司注册本身要 1-2 周
  • 对公账户 + 银行 U 盾要 1-2 周
  • 小程序企业认证(账号申请 + 微信认证 300 元/年)要 1-2 周
  • ICP 备案要 7-20 个工作日
  • 总共 4-6 周走完。如果 Week 5 启动,Week 11 上线刚好

公司注册形式仍按 v2.1:普通有限公司 + 章程自约束(不分红、利润转工资、解散归社会)。

五、合规专题:ICP 备案

5.1 ICP 备案到底是什么

中国法律要求:任何面向国内用户提供网站/小程序服务的主体,在工信部完成"非经营性互联网信息服务备案"(简称 ICP 备案)。没备案的境内网站 80 端口会在 24-48 小时内被切断,小程序无法发布。

5.2 备案要求

项目个人备案企业备案
主体材料身份证营业执照 + 法人身份证
服务器必须国内云(阿里/腾讯/华为/百度等)同左
域名已实名认证的 .cn/.com 等同左
内容限制不能做经营性:电商、互动社区、知识付费、教育培训、医疗资讯等几乎都被禁按公司经营范围,大部分都可
时间7-20 工作日同左
对本项目适用度❌ 内容限制基本封死✅

5.3 试点期备案策略

  1. Week 1-4 田野期:用 Surge / Vercel / Cloudflare Pages 部署原型,自己人和早期访谈对象用,不公开宣传。无需备案
  2. Week 5:启动公司注册
  3. Week 6-9:边开发边走备案。开发用境外服务器(Vercel 等),备案下来切换到境内
  4. Week 10+:备案完成,小程序提审,正式上线

5.4 备案的隐藏雷

  • 注册地址不一致:营业执照地址 vs 实际办公地址,备案审核会查。建议公司地址用真实办公地址(哪怕是租的"虚拟办公室")
  • 负责人手持身份证拍照:大部分省份现在要求"备案专用幕布 + 手持身份证视频"。提前做好
  • 名称包含"教育"、"医疗"、"金融"、"新闻"等敏感词:触发额外前置审批(教育部门、卫健委等),时间翻倍。这就是 v2.1 §3.3 漂白命名的现实根据——"数字素养服务中心"不触发,"工友教育"会触发
  • 小程序专项备案:小程序在 ICP 主备案之外,还要在微信生态内单独备案。两个流程并行

六、合规专题:数据保护

6.1 适用法律(扫一遍)

  • 《个人信息保护法》(PIPL)(2021):个人信息收集、使用、跨境传输的总章
  • 《数据安全法》(2021):数据分级、重要数据出境
  • 《网络安全法》(2017):网络运营者义务、关键信息基础设施保护
  • 《未成年人保护法》(2021):14 岁以下儿童信息特别保护

6.2 试点期的最小化合规配置

原则

最小化收集 + 明示授权 + 国内存储 + 加密传输。这四条做到,试点期(50 用户内)绝大多数风险被覆盖。

具体到字段层面

字段建议风险
微信 OpenID✅ 必收(用作 user ID)低
用户昵称 + 头像✅ 微信授权获取低
手机号⚠ 非必要不收,真要收需短信验证 + 明示授权中(属敏感个人信息)
身份证号❌ 试点期不收极高
位置(精确)⚠ 用户每次主动授权,不持久保存中
位置(模糊,如城市)✅ 可保存,但要明示低
支付信息由微信支付/支付宝处理,你的服务不接触卡号由平台兜底
儿童信息(<14)❌ 试点期完全不收,避免 PIPL 第 31 条极高
医疗/健康❌ 试点期完全不收极高
聊天/AI 对话✅ 加密存储,默认 90 天后删除低

6.3 必须做的最小动作

  1. 《用户协议》和《隐私政策》:小程序上线必备。可参考开源模板(GitHub 上有大量中国合规版本),自己改一下
  2. 首次启动弹窗:用户同意《隐私政策》才能继续。这是 PIPL 强制要求
  3. 数据存储位置:全部在中国大陆服务器(阿里云 / 腾讯云的国内 region)
  4. 加密传输:HTTPS,小程序也强制 HTTPS
  5. 用户数据导出 + 删除接口:PIPL 给用户的权利。试点期可以做"用户提交工单后人工处理",不需要自动化
  6. 数据访问日志:谁在什么时候访问了谁的数据,留 6 个月

6.4 跨境数据问题(特别提醒)

关键合规点

试点期如果你把用户数据发到境外(包括但不限于:用 OpenAI / Claude 的 API 跑用户聊天、用 Vercel 等境外服务存数据、用 Notion / Airtable 等境外 SaaS 做用户管理),属于"个人信息跨境传输",理论上需要走 PIPL 第 38 条规定的安全评估。

试点期实操规避:

  • AI 调用全部走国内 LLM API(DeepSeek / 通义 / 文心 / 豆包)
  • 用户数据不用境外 SaaS。CRM 用国内的(石墨/飞书),分析工具用国内的(神策的免费版/友盟+)
  • 团队协作工具(代码 GitHub / 文档)不涉及用户数据就 OK

6.5 数据合规的诚实声明

上面的策略是试点期(50 用户、低敏感度数据)的最小合规配置。如果产品方向走向以下任何一个,合规复杂度会指数级上升,需要专门请律师:

  • 涉及医疗、健康、心理咨询类信息
  • 涉及 14 岁以下儿童(母婴产品里的婴儿信息)
  • 涉及金融/借贷/保险
  • 用户数据规模超过 100 万(触发 PIPL 关键信息基础设施)
  • 涉及定位、人脸识别、生物特征

这些复杂度本身就是产品方向选择的负权重——同样有商业价值的方向,合规越轻的越优先做。

八、决策点与失败应变

8.1 三个判断点(显式写出)

8.2 阶段性失败的几种情况

失败 A:Week 5 找不到主方向

意味着田野结果太分散,没有任何一个轨道收敛到清晰痛点。应变:Week 5-6 多花 2 周再做一轮田野(把田野期延长到 7 周,Phase 2 压缩到 3 周)。仍然找不到,意味着团队对目标用户群理解不够——这是一个项目级警告,不是产品级失败。

失败 B:Week 9 备案/小程序审核没过

常见原因:类目选错、内容触发审查、企业资质问题。应变:H5 + 公众号先上(微信生态内无需 ICP 备案的边角通道)。把小程序当作下一个版本的目标。备案问题不影响产品迭代。

失败 C:Week 12 留存 < 30%

意味着产品有人用但留不住。应变:Week 13 复盘聚焦"为什么不回来"——访谈每一个流失用户。常见原因:解决的不是 P0 痛点 / 替代方案太多 / 体验有硬伤。决定是pivot(改产品方向)而非 kill。

失败 D:50 用户里没有 20 个第一二产业

意味着传播渠道走偏了——自然增长把你引到错的人群里。应变:这是 v2.1 §5.1 强调的"种子用户成分锁死"问题。Week 11-12 必须人为干预——把传播流量重新引到目标人群。如果实在做不到,意味着产品包装/语言/触达渠道根本上不对工友胃口,Week 13 复盘必须严肃对待。

8.3 90 天决策矩阵

留存用户成分付费/数据贡献意愿决策
≥ 30%≥ 20 个目标人群≥ 10 个用户愿意付 ≥ 10 元/月✅ Continue - 进入下一个 90 天
≥ 30%< 20 个目标人群有付费意愿⚠ Pivot (用户) - 产品 OK,但用户群错了,重新定位
< 30%≥ 20 个目标人群有意愿⚠ Pivot (产品) - 用户对,但产品没解决他们最痛的

提示

kill 不是项目失败。kill 是 90 天投资了 ~20 万人民币 + 三人精力,买到了一个清晰的"这条路走不通"的判断。比"继续耗下去"省钱也省时间。项目级别的失败信号是连续 2-3 个 90 天都 kill,意味着团队对市场的判断力有问题——那时再考虑根本性调整。
